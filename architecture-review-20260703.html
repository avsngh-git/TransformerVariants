Deep: small interface, large implementation

Problem: The evaluation pipeline's interface (its ~30 import dependencies, 400-line main orchestrating 9 steps) is nearly as complex as its implementation — a textbook shallow module.

Solution: Absorb orchestration into a single EvaluationPipeline module with one public method — run(checkpoints, output_dir, device, data_dir) — and internal methods for each phase.

  • Leverage: CLI, tests, and future dashboard share one interface
  • Locality: Error handling, seed grouping, step ordering concentrate inside
  • Interface shrinks: 1 method replaces 30 function calls + manual wiring
  • Testable: Integration test becomes pipeline.run(debug_checkpoints)
  • Deletion test: Deleting the script reappears as wiring across every caller

Each VariantSpec knows how to configure itself. build() is a 10-line dispatcher.

Problem: build() leaks knowledge of every variant's construction needs via if attention_type == ... chains — adding a variant means editing the generic builder.

Solution: Give VariantSpec a prepare_config(dims: dict) → ModelConfig method that each variant overrides to set its own window_size, projection_rank, n_kv_head, and per_layer_configs.

  • Locality: Variant-specific knowledge concentrates with its VariantSpec
  • Leverage: build() shrinks to a generic 10-line dispatcher
  • Open/closed: New variants require zero edits to build()
  • Testable: Each spec's prepare_config is independently testable

After

ProbeTarget (protocol)

forward(x) → logits

get_layer_outputs(x) → list[Tensor]

get_attention_weights(x) → list[Tensor] | None

config: ModelConfig

Probes take ProbeTarget — no model.blocks, no hook registration

Adapter on ModernTransformer / VanillaTransformer implements ProbeTarget

One seam. Two adapters (real model + test fake). Real variation justifies the seam.

Problem: Probes reach into model internals (.blocks, .attn.attn_dropout, hook registration) — testing requires fully constructing every variant.

Solution: Define a ProbeTarget protocol with get_layer_outputs() and get_attention_weights(); adapters on each transformer class implement it, and a simple fake satisfies tests.

  • Locality: Hook registration logic concentrates in the adapter, not in every probe
  • Leverage: New probes only need ProbeTarget, not model archaeology
  • Testable: Fake ProbeTarget with canned tensors — no GPU, no model
  • Two adapters justify the seam: real model adapter + test fake

One protocol. All attention modules satisfy it. Cache is Optional[KVCache].

Problem: Understanding "attention" requires reading 6 files with 3 parallel inheritance roots and incompatible KV-cache contracts.

Solution: Formalize the existing informal contract — forward(x, kv_cache=None) → (output, new_kv_cache) — as a typed Protocol. Unify KV-cache into a single KVCache dataclass (or None for V5).

  • Leverage: TransformerBlock, probes, and generation code all program to one interface
  • Locality: Cache semantics defined once, not rediscovered per variant
  • AI-navigable: One protocol definition is the index into all attention variants

Open question in CONTEXT.md: "KV-Cache unification — deferred until all variants are implemented." All variants now exist. The friction is real — but ADR 0001 (SWA parameterizes FlashAttention) and ADR 0002 (V5 no KV-cache) should be respected. A protocol that allows None cache (V5) and variant-specific cache internals (tuple vs pre-allocated) satisfies both ADRs without contradicting them.

Top Recommendation

Candidate 1: Deepen the Evaluation Pipeline — the highest-leverage opportunity. Today, scripts/evaluate.py is a 400-line shallow orchestrator that any caller (tests, future dashboard, notebook) must replicate. Absorbing it into a deep EvaluationPipeline module gives one interface for all callers, concentrates error handling and step ordering inside, and immediately simplifies integration testing to a single pipeline.run() call. No ADR conflicts, no risk — pure deepening.
